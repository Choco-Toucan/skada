### #1 LeaderboardMapper.update() 缺少 status 和 current_instance_id 列

**根因**: `LeaderboardMapper.xml` 的 UPDATE 语句没有包含 `status` 和 `current_instance_id` 字段，导致 `stop()` 和 `roll()` 方法中对这两个字段的 `set` 操作不会持久化到数据库。

**修复文件**: `mng-service/src/main/resources/mapper/LeaderboardMapper.xml`

**修复方案**: 在 UPDATE 语句的 SET 子句中增加两列：

```xml
<update id="update">
    UPDATE leaderboard_plan
    SET name = #{name},
        start_time = #{startTime},
        end_time = #{endTime},
        max_query_users = #{maxQueryUsers},
        allow_duplicate_report = #{allowDuplicateReport},
        allow_history_query = #{allowHistoryQuery},
        roll_strategy = #{rollStrategy},
        roll_interval_value = #{rollIntervalValue},
        roll_interval_unit = #{rollIntervalUnit},
        roll_user_count = #{rollUserCount},
        status = #{status},
        current_instance_id = #{currentInstanceId},
        update_by = #{updateBy}
    WHERE id = #{id}
</update>
```

**影响范围**: 所有调用 `leaderboardMapper.update(lb)` 且设置了 `status` 或 `currentInstanceId` 的路径：
- `LeaderboardConfigService.stop()` — 终止操作
- `LeaderboardConfigService.roll()` — 手动滚动
- `LeaderboardRollScheduler.stopLeaderboard()` — 调度器自动终止
- `LeaderboardRollScheduler.rollLeaderboard()` — 调度器自动滚动
- `ScoreService.doRoll()` — api-service 中的用户数滚动

**验证方式**: 创建排行榜后调用 stop 接口，查询数据库确认 status 变为 `stopped`。

### #5 LogInterceptor 将 secretKey 和密码写入明文日志

**根因**: `LogInterceptor.afterCompletion()` 将请求 body 和响应 body 原样写入 `api.log`，包含敏感字段。

**修复文件**: `common/src/main/java/com/skada/common/interceptor/LogInterceptor.java`

**修复方案**: 在 `getRequestBody()` 和 `getResponseBody()` 方法中对敏感字段做脱敏处理。定义脱敏字段名列表，用正则替换其值为 `***`：

```java
private static final Set<String> SENSITIVE_FIELDS = Set.of(
    "secretKey", "password", "token", "secret_key"
);

private String maskSensitive(String body) {
    if (body == null || body.isEmpty()) return body;
    String masked = body;
    for (String field : SENSITIVE_FIELDS) {
        // 匹配 JSON 中的 "field":"value" 或 "field":"value"
        masked = masked.replaceAll(
            "(?i)\"" + field + "\"\\s*:\\s*\"[^\"]*\"",
            "\"" + field + "\":\"***\"");
    }
    return masked;
}
```

然后在 `afterCompletion()` 中对 params 和 responseBody 调用 `maskSensitive()`。

**验证方式**: 调用登录接口和分数上报接口，检查 `logs/api.log` 中 `password` 和 `secretKey` 是否被脱敏。

### #6 lb.getMaxQueryUsers() 和 lb.getAllowDuplicateReport() 可能 NPE

**根因**: `Leaderboard` 模型中这两个字段声明为 `Integer`，自动拆箱时如果值为 null 会抛出 NPE。当前各调用方各自处理空值，分散且容易遗漏。

**设计原则**: 由 model 自身的 getter 处理数据库字段的默认值，统一收敛空值处理逻辑，外部调用方无需关心 null 问题。

**修复文件**:
1. `mng-service/src/main/java/com/skada/mng/model/Leaderboard.java` — getter 返回默认值
2. `api-service/src/main/java/com/skada/api/model/Leaderboard.java` — 同步修改（两个服务各有独立的 Leaderboard 模型）

**修复方案**: 修改 getter，当字段为 null 时返回 schema 定义的默认值：

```java
// maxQueryUsers: schema DEFAULT 1000
public Integer getMaxQueryUsers() {
    return maxQueryUsers != null ? maxQueryUsers : 1000;
}

// allowDuplicateReport: schema DEFAULT 0
public Integer getAllowDuplicateReport() {
    return allowDuplicateReport != null ? allowDuplicateReport : 0;
}

// allowHistoryQuery: schema DEFAULT 1
public Integer getAllowHistoryQuery() {
    return allowHistoryQuery != null ? allowHistoryQuery : 1;
}
```

同时将字段声明改为 `private Integer maxQueryUsers` 等（保持 Integer 接收数据库值），仅在 getter 出口做拆箱保护。

**注意**: MyBatis 通过 setter 或反射直接赋值，getter 中的默认值逻辑对 MyBatis 映射无影响，仅影响业务代码调用。

**验证方式**: 手动将数据库中某排行榜的 `max_query_users` 或 `allow_duplicate_report` 设为 NULL，调用查询/上报接口确认不抛 NPE。

### #7 mng-service 手动滚动与调度器滚动之间无互斥

**根因**: `LeaderboardConfigService.roll()` 和 `LeaderboardRollScheduler.rollLeaderboard()` 两个入口对同一排行榜执行滚动时没有锁保护。

**修复文件**:
1. `mng-service/src/main/java/com/skada/mng/service/LeaderboardConfigService.java`
2. `mng-service/src/main/java/com/skada/mng/scheduler/LeaderboardRollScheduler.java`

**修复方案**: 在两个滚动方法中都使用 DistributedLock，锁的 key 为 `roll:leaderboard:{leaderboardId}`，与 `ScoreService.doRoll()` 中用户数滚动使用的锁模式保持一致：

`LeaderboardConfigService.roll()` 方法包裹分布式锁：
```java
String lockValue = UUID.randomUUID().toString();
String lockName = "roll:leaderboard:" + leaderboardId;
if (!distributedLock.tryLock(lockName, lockValue, 10, TimeUnit.SECONDS)) {
    throw new BusinessException("排行榜正在滚动中，请稍后重试");
}
try {
    // ... 现有滚动逻辑 ...
} finally {
    distributedLock.unlock(lockName, lockValue);
}
```

`LeaderboardRollScheduler.rollLeaderboard()` 同理。

注意：需要在 `LeaderboardConfigService` 中注入 `DistributedLock`（当前未注入）。

**验证方式**: 单元测试模拟并发滚动调用，验证只有一个成功。
